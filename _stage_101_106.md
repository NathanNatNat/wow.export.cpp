## File: src/js/components/checkboxlist.cpp (audit entry #101)

FINDING: `scrollIndex()` clamps incorrectly — can return a negative value before the caller clamps it
  JS Source: src/js/components/checkboxlist.js lines 67–69
  Details: The JS `scrollIndex` computed property returns `Math.round((items.length - slotCount) * scrollRel)` and never goes negative because `scrollRel` is clamped to [0,1] and `items.length - slotCount` is the range. However the C++ `scrollIndex()` static function returns the raw rounded value, which can be negative when `items.size() < state.slotCount`. The caller then applies `std::max(0, idx)` at line 165, which handles the underflow — but the returned value from `scrollIndex()` itself can still be negative, meaning the function's contract differs from what the JS property computes. This is a minor deviation; no user-visible bug because the caller guards against it, but the internal function is not a faithful 1-to-1 match.

FINDING: `wheelMouse()` uses `itemHeight` parameter where JS uses `child.clientHeight` at runtime
  JS Source: src/js/components/checkboxlist.js lines 140–149
  Details: In the JS `wheelMouse`, the child element's `clientHeight` is measured at runtime from the DOM (`const child = this.$el.querySelector('.item')`). The C++ version passes `itemHeight` (computed as `max(frameHeightWithSpacing, 26.0f)`) and checks `itemHeight > 0.0f`. The JS also checks `child !== null` before proceeding. The C++ additionally checks `!items.empty() && state.slotCount > 0`, which the JS does not (JS relies on `child !== null`). The C++ `scrollCount` calculation uses `containerHeight / itemHeight` whereas the JS uses `this.$el.clientHeight / child.clientHeight`. This is functionally correct but the added empty/slotCount guard is an addition not present in JS.

FINDING: `wheelMouse()` direction logic is inverted relative to JS
  JS Source: src/js/components/checkboxlist.js lines 146
  Details: In JS, `direction = e.deltaY > 0 ? 1 : -1`. In the C++ `render()` at line 177–179, `wheel` is obtained from `ImGui::GetIO().MouseWheel` and passed as `-wheel` to `wheelMouse()`. Inside `wheelMouse`, the C++ condition is `direction = wheelDelta > 0 ? 1.0f : -1.0f`. Since ImGui's `MouseWheel` is positive when scrolling up, negating it means a positive `wheelDelta` into `wheelMouse` means scrolling down — matching JS `deltaY > 0` = scroll down. Functionally equivalent, but the negation at the call site (`-wheel`) rather than inside the function is a structural difference from the JS where the raw `deltaY` is used directly.

FINDING: Scrollbar thumb is rendered with `ImDrawList::AddRectFilled` (raw draw call) instead of a native ImGui widget
  JS Source: src/js/components/checkboxlist.js lines 170–171 (template, `.scroller` div)
  Details: Per CLAUDE.md, `ImDrawList` calls should be reserved for effects with no native equivalent. A scrollbar thumb does have a native ImGui equivalent (`ImGui::BeginChild` with built-in scrollbar). The custom scrollbar is rendered entirely with raw `AddRectFilled`. However since this is a custom virtual-scrolling scrollbar matching the exact JS layout (where CSS positions a `.scroller` div), and the JS component's scrollbar is not a native browser scrollbar, this deviation may be acceptable. Still worth flagging.

FINDING: `selected` CSS class on items is not reflected — selected rows are not visually distinguished
  JS Source: src/js/components/checkboxlist.js line 172
  Details: The JS template has `:class="{ selected: item.checked }"` on each `.item` div, which applies a `selected` CSS class when the checkbox is checked, giving a visual highlight. The C++ rendering loop (lines 236–260) does not apply any background highlight or color change when `checked` is true. Only the checkbox widget itself reflects the checked state. The row background is not changed.
