FINDING: `loadSchema()` does not use the CASC cache object (`casc.cache`) for DBD retrieval, unlike the JS original
  JS Source: src/js/db/DBCReader.js lines 177‚Äì194
  Details: The JS `loadSchema()` uses `core.view.casc?.cache` (a `BuildCache` object) with `cache.getFile(cache_key, constants.CACHE.DIR_DBD)` to retrieve cached DBDs and `cache.storeFile(cache_key, raw_dbd, constants.CACHE.DIR_DBD)` to persist new downloads. If `cache` is null/undefined it skips caching. The C++ version (`DBCReader.cpp` lines 189‚Äì224) bypasses this entirely ‚Äî it always uses `std::filesystem::exists(dbdCachePath)` + `BufferWrapper::readFile` for reading and `raw_dbd.writeToFile(dbdCachePath)` for writing. By contrast `WDCReader.cpp` correctly implements `getActiveCascCache()` and calls `cache->getFile()` / `cache->storeFile()`, falling back to direct filesystem only when cache is null. `DBCReader.cpp` should be updated to use the same `getActiveCascCache()` pattern that `WDCReader.cpp` uses.

FINDING: `parse()` and `loadSchema()` are declared synchronous (`void`) instead of `async` ‚Äî the callers must not `co_await` them, but no async-equivalent wrapper exists
  JS Source: src/js/db/DBCReader.js lines 162, 244
  Details: Both `loadSchema()` and `parse()` are `async` in JS, returning Promises. This matters because `loadSchema` performs `await dbd_manifest.prepareManifest()`, `await cache.getFile(...)`, `await generics.downloadFile(...)`, and `await cache.storeFile(...)`. The C++ makes all of these synchronous calls, which aligns with the rest of the codebase's synchronous approach and the synchronous implementations of `prepareManifest()`, `downloadFile()`, etc. This is an acceptable deviation per project patterns, but should be documented as such.

FINDING: `_read_field()` missing `FieldType::Int64` and `FieldType::UInt64` switch cases
  JS Source: src/js/db/DBCReader.js lines 390‚Äì408
  Details: The JS `_read_field` switch handles `FieldType.Int8`, `FieldType.UInt8`, `FieldType.Int16`, `FieldType.UInt16`, `FieldType.Int32`, `FieldType.UInt32`, `FieldType.Float`, and a default. The JS is equally incomplete here; however DBC files (pre-Cataclysm) don't have 64-bit fields in practice.

FINDING: `_read_field_array()` returns typed homogeneous vectors in C++ vs a mixed-type JS Array in JS
  JS Source: src/js/db/DBCReader.js lines 417‚Äì423
  Details: In JS, `_read_field_array` returns a plain `Array` that stores whatever `_read_field` returns (a JS value), stored directly as `out[name]`. In C++, `_read_record` (lines 410‚Äì443) post-processes the `vector<FieldValue>` result into type-specific vectors (`vector<std::string>`, `vector<float>`, `vector<int64_t>`, `vector<uint64_t>`). This is a necessary C++ adaptation due to the type system, but if `values` is non-empty and the first element holds none of the four checked types, the field is silently dropped from `out` (no `else` branch stores anything). This diverges from the JS, which always stores the array. An `else` fallback should store an empty or raw vector.
