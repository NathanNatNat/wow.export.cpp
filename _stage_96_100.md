## File: src/js/casc/realmlist.cpp (audit entry #96)

FINDING: JS `load()` validates that the URL is actually a string and throws if not, but C++ silently produces "undefined" and continues
  JS Source: src/js/casc/realmlist.js lines 39–41
  Details: JS does `let url = String(core.view.config.realmListURL)` and then `if (typeof url !== 'string') throw new Error('Missing/malformed realmListURL in configuration!')`. While `String()` always returns a string (making the typeof check practically unreachable), the intent is to throw if the config value is missing/malformed. The C++ `jsStringCoerce` returns "undefined" (the string) for a null pointer and then silently proceeds to use it as a URL. This deviates from the error-throwing intent expressed in the JS, even if JS's own check is unreachable in practice.

FINDING: The `jsStringCoerce` helper is large and unnecessary for this use case
  JS Source: src/js/casc/realmlist.js lines 39
  Details: JS simply calls `String(core.view.config.realmListURL)` — a built-in coercion. The C++ implements a full JS-style string coercion helper (`jsStringCoerce`) that handles null, boolean, number, float, array, and object cases. For a config URL value (expected to be a string or undefined), this is far more complex than the JS idiom. The complexity is not wrong per se, but the unusual-path handling (NaN, Infinity, arrays, objects) is untested dead code that adds maintenance burden. A simpler `std::string` extraction with a nullopt/empty check would be more idiomatic and faithful.

FINDING: `loadAsync()` is not part of the original JS module exports
  JS Source: src/js/casc/realmlist.js lines 67–69
  Details: JS exports only `{ load }`. The C++ header and implementation also expose `loadAsync()`. This is an acceptable C++ adaptation to support async loading patterns used elsewhere in the codebase, but it is a deviation from the original JS API surface.

## File: src/js/casc/salsa20.cpp (audit entry #97)

FINDING: C++ constructor key-length validation checks hex-string length but error message says "16 or 32 bytes"
  JS Source: src/js/casc/salsa20.js lines 24–25
  Details: JS: `if (key.length !== 32 && key.length !== 64) throw new Error('Unexpected key length. 16 or 32 bytes expected, got ' + key.length)`. The error message in C++ says "16 or 32 bytes expected, got " + key.size(). But `key.size()` is the number of hex characters (32 or 64), not bytes. JS also reports the character count (since `key.length` is the hex string length), so both messages are equally misleading — this matches JS faithfully.

## File: src/js/casc/vp9-avi-demuxer.cpp (audit entry #100)

FINDING: The TODO 207 comment in `find_chunk()` is incorrect; the C++ loop bound already matches JS

  Details: JS: `for (let i = 0; i < data.length - 4; i++)`. C++ (line 78): `for (size_t i = 0; i + 4 < data.size(); i++)`. These are mathematically identical: `i + 4 < size` ↔ `i < size - 4`. The TODO comment falsely claims they differ and instructs a fix that is already done. The TODO 207 comment should be removed as it is misleading.

FINDING: JS `extract_frames()` is an `async generator` (`async*`); C++ converts it to a callback-based `void extract_frames(callback)`
  JS Source: src/js/casc/vp9-avi-demuxer.js lines 83–126
  Details: JS callers use `for await (const frame of demuxer.extract_frames())` to consume frames lazily. C++ uses a synchronous callback `std::function<void(const FrameInfo&)>`. This is a structural adaptation required by C++ (no native async generators), but it means callers must restructure how they consume frames. This is a known and acceptable deviation, but it should be documented.

FINDING: JS `extract_frames()` uses `console.log` for debug output; C++ uses `logging::write`
  JS Source: src/js/casc/vp9-avi-demuxer.js lines 108, 125
  Details: JS uses `console.log(...)` for block/frame parsing diagnostics inside `extract_frames`. C++ uses `logging::write(...)`. This is an acceptable adaptation (the JS app does not route `console.log` through the app's log system, while C++ does). Functionally the debug output is equivalent.

FINDING: JS `parse_movi_frames_with_remainder` returns `{ frames: [], remainder: null }` when movi not found; C++ returns `ParseResult` with empty frames and `hasRemainder = false`
  JS Source: src/js/casc/vp9-avi-demuxer.js lines 141–149

FINDING: C++ `find_chunk()` signature accepts `const char fourcc[4]` (decays to `const char*`) instead of a `std::string` or `std::string_view`
  JS Source: src/js/casc/vp9-avi-demuxer.js lines 67–77
  Details: In JS, `find_chunk` accepts a string argument.
